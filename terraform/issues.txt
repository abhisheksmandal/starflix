[ ] H1. Frontend ECS health check calls `curl`, which is not in the frontend image
--------------------------------------------------------------------------------
WHERE:
  - modules/ecs-service/main.tf:69 (container healthCheck command)
  - frontend/Dockerfile (base: nginx:1.27-alpine — no curl installed)
  - backend/Dockerfile:4 (RUN apk add --no-cache curl — proof the pattern was known)

WHAT:
  The ecs-service module hardcodes, for EVERY service:
      command = ["CMD-SHELL", "curl -f http://localhost:${var.container_port}/health || exit 1"]
  The backend image installs curl; the frontend image (nginx:1.27-alpine) does not.

WHY IT BREAKS:
  On the frontend task, `curl` is not found (exit 127) -> `|| exit 1` -> container
  reported UNHEALTHY. The container is essential=true, so ECS repeatedly kills and
  replaces the task; the frontend service never reaches steady state.
  (The /health PATH is fine — nginx SPA fallback try_files -> /index.html returns 200.
   The ONLY failure is the missing curl binary.)

FIX (pick one):
  Option A (simplest, mirrors backend): add to frontend/Dockerfile final stage:
      RUN apk add --no-cache curl
  Option B (no image change): switch the health check to BusyBox wget, which ships
    in the alpine image. In modules/ecs-service/main.tf:
      command = ["CMD-SHELL", "wget -q --spider http://localhost:${var.container_port}/health || exit 1"]
    NOTE: if switching to wget, verify backend also has wget (or keep curl there).
  Option C (most flexible): make the command/path a module variable
    (e.g. var.health_check_command) and pass a curl variant for backend, wget for
    frontend. Preferred long-term.
  RECOMMENDED: Option A now (unblocks deploy); consider Option C later.

[ ] M1. enable_https=true with empty/placeholder cert ARNs passes plan but fails apply
--------------------------------------------------------------------------------
WHERE:
  - environments/dev/terraform.tfvars (frontend_acm_certificate_arn / backend_acm_certificate_arn
    currently hold placeholder ARNs: ...000000000000... and ...111111111111...)
  - modules/alb/main.tf:109, 203 (certificate_arn on the HTTPS listeners)
  - environments/dev/variables.tf:121-131

WHAT:
  AWS rejects an HTTPS listener whose certificate_arn is empty or points to a
  non-existent certificate. Plan does not catch this; apply will fail.

WHY:
  No precondition/validation asserts the ARN is real when enable_https is true.

FIX:
  1. Populate the two ARN vars with REAL issued ACM cert ARNs (ap-south-1) before apply.
  2. Add a guardrail so a bad config fails at plan, not apply. Options:
     - A `check` block in environments/dev, or
     - A `precondition` on the ALB HTTPS listeners in modules/alb/main.tf:
         lifecycle {
           precondition {
             condition     = !var.enable_https || (var.acm_certificate_arn != null && var.acm_certificate_arn != "")
             error_message = "acm_certificate_arn is required when enable_https is true."
           }
         }
       (same for backend_acm_certificate_arn on the backend HTTPS listener)


[ ] M2. Backend API served in plaintext to the whole internet on :4000, even with HTTPS on
--------------------------------------------------------------------------------
WHERE:
  - modules/alb/main.tf:175-189 (backend_http listener forwards, never redirects)
  - modules/security-groups/main.tf:32-42 (alb_backend_http ingress: 4000 from 0.0.0.0/0)

WHAT:
  With HTTPS now the intended path (public_backend_url = https://...), the
  unencrypted :4000 endpoint remains publicly reachable and forwards to the API.

WHY IT MATTERS:
  Public plaintext API surface. Comments call it intentional, but once clients are
  on HTTPS it should be closed or redirected.

FIX (pick one, once HTTPS clients are confirmed):
  - Redirect 4000 -> 443 (make the backend_http listener a redirect action, mirroring
    the frontend_http dynamic redirect when var.enable_https is true), OR
  - Remove the aws_vpc_security_group_ingress_rule.alb_backend_http rule so :4000 is
    no longer internet-reachable, OR
  - Gate both behind a variable (e.g. var.expose_backend_http) defaulting to false.

--------------------------------------------------------------------------------
WHERE:
  - environments/dev/variables.tf:45-54 (validation only checks >= 1)
  - modules/vpc/main.tf:41-52 (private subnets index var.azs[count.index])
  - environments/dev/locals.tf:11-15 (azs sliced to length(public_subnet_cidrs))

WHAT:

  Add a validation on private_subnet_cidrs:

      condition     = length(var.private_subnet_cidrs) == length(var.public_subnet_cidrs)
      error_message = "private_subnet_cidrs must have the same number of entries as public_subnet_cidrs."
    }


[ ] L3. Container health-check path hardcoded (/health) while ALB paths are configurable
--------------------------------------------------------------------------------
WHERE:
  - modules/ecs-service/main.tf:69 (hardcoded /health)
  - modules/alb/variables.tf:56-66 (health_check_path_frontend="/", health_check_path_backend="/health")

WHAT:
  Frontend ALB target group checks "/" but the container checks "/health". Today both
  pass only because nginx's SPA catch-all returns index.html (200) for /health.

WHY IT MATTERS:
  Latent coupling — a future `location /health { return 404; }` in nginx would silently
  crash-loop the frontend.

FIX:
  Parameterize the container health-check path (and command) in ecs-service as a variable,
  and pass the same path used for the ALB target group per service. (Combines well with H1 Option C.)
